# Yomoyo ‚Äî Development Guidance for Claude Code

## Firebase Architecture Rules

### Guiding principle
Prefer the Firebase JS SDK (`firebase` npm package) for all data access.
Use native RNFirebase modules only where the native OS layer is genuinely required.

### What uses native RNFirebase (do not change without explicit approval)
- `@react-native-firebase/app` ‚Äî native Firebase SDK bootstrap
- `@react-native-firebase/auth` ‚Äî required for Google Sign-In and Apple Sign-In credential exchange
- `@react-native-google-signin/google-signin` ‚Äî native Google Sign-In OS flow
- `@invertase/react-native-apple-authentication` ‚Äî native Apple Sign-In OS API

### What must NOT use RNFirebase native modules
- Firestore ‚Äî uses `firebase/firestore` from the JS SDK (`src/lib/books/readingActivity.ts`)
- Any future data access layer ‚Äî default to the JS SDK

### Do not add new RNFirebase native modules without explicit approval
Adding a new `@react-native-firebase/*` package (Storage, Analytics, Messaging, etc.)
requires explicit decision. Each one potentially reintroduces iOS build fragility.
Default answer: use the Firebase JS SDK equivalent, or avoid the feature.

## iOS Native Integration Rules

### All iOS compatibility fixes must live in `plugins/`
Never hand-edit `ios/Podfile` directly ‚Äî it is regenerated by `expo prebuild`.
Patches in `plugins/` survive regeneration; hand edits do not.

### Current active patches
- `plugins/withFmtConsteval.js` ‚Äî Clang 17 / fmt 11.x consteval workaround (Xcode 26.4)
- `plugins/withFirebaseStaticFramework.js` ‚Äî RNFBApp/RNFBAuth static framework alignment
  and `CLANG_ALLOW_NON_MODULAR_INCLUDES_IN_FRAMEWORK_MODULES` for those targets

### `useFrameworks: "static"` policy
Currently set in `expo-build-properties` (`app.json`). Kept because it is required for
RNFBApp/RNFBAuth static linking. Evaluate removing it after confirming no other native
module depends on it. If it is removed, `plugins/withFirebaseStaticFramework.js` and
the `$RNFirebaseAsStaticFramework` global can also be removed.

Do not add new native pods that require static framework workarounds without explicit
approval.

## Firebase JS SDK Initialization

The Firebase JS SDK is initialized in `src/lib/firebase/index.ts` and imported at the
app root (`App.tsx`). It reads from `EXPO_PUBLIC_FIREBASE_*` environment variables.

Required `.env` variables (values from `GoogleService-Info.plist`):
```
EXPO_PUBLIC_FIREBASE_API_KEY=
EXPO_PUBLIC_FIREBASE_PROJECT_ID=
EXPO_PUBLIC_FIREBASE_STORAGE_BUCKET=
EXPO_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
EXPO_PUBLIC_FIREBASE_APP_ID=
```

## App Config and Bundle Identifiers

`app.config.ts` overrides `app.json` with env-driven bundle identifiers:
- `IOS_BUNDLE_IDENTIFIER` ‚Üí `ios.bundleIdentifier` (fallback: `com.furugenisland.yomoyo`)
- `ANDROID_PACKAGE_NAME` ‚Üí `android.package` (fallback: `com.furugenisland.yomoyo`)

All other shared config (plugins, Firebase setup, splash, icons) stays in `app.json`.
Do not add developer-specific values to `app.json`. Add them as env vars in `.env.example`.

---

## Test Setup

`firebase/firestore` and `firebase/app` are mapped to manual mocks via `moduleNameMapper`
in `package.json`. Tests do not need `firebase` installed to run ‚Äî the mocks are used
automatically by Jest. When adding new firebase features, add corresponding exports to
`__mocks__/firebase/firestore.js` or create a new mock file as needed.
